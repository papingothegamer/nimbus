# Daily Audit: July 23, 2026

## Focus: Track Grouping — Duplicated Clip Not Rendering in Timeline

### Problem Statement
When a track is duplicated within a track group (or when two tracks are grouped after duplication), the second track's clip waveform does not appear in the timeline grid. The audio plays correctly from both tracks, the mixer shows two channels with VU activity, and the track header is visible — but the `ClipComponent` in the second track's lane is invisible.

A red-border debug test on July 20 confirmed that the `ClipComponent` either does not exist, or has zero/invalid bounds where the second clip should render.

### Root Cause Analysis

After tracing through the complete data flow from `TimelineProject::duplicateTrack()` → `insertTrack()` → `trackAdded()` → `addClipToTrack()` → `trackClipsChanged()` → `TrackLaneComponent::updateClips()`, the most probable root causes are:

#### Cause 1: `TrackLaneComponent::setTrackIndex()` does not call `updateClips()`
When the `tracksGrouped()` nuclear rebuild fires, or when `trackAdded()` shifts indices of existing lanes, `setTrackIndex()` is called on each lane — but it **only updates the integer field**:
```cpp
void TrackLaneComponent::setTrackIndex(int newIndex) {
    trackIndex = newIndex;
    // NO call to updateClips()!
}
```
If a lane's `trackIndex` changes, it will continue to show stale clips from the old index. This means the lane might be looking at wrong data, or have zero ClipComponents from its initial construction, and never get refreshed.

#### Cause 2: Listener callback ordering — `trackClipsChanged` may fire before `trackAdded`
In `duplicateTrack()`:
```cpp
insertTrack(trackIndex + 1, newTrack);  // fires trackAdded → creates lane
auto originalClips = getClipsOnTrack(trackIndex);
for (const auto& clip : originalClips) {
    addClipToTrack(trackIndex + 1, clip->clone());  // fires trackClipsChanged
}
```
The `addClipToTrack` fires `trackClipsChanged(trackIndex + 1)`. At this point, the new `TrackLaneComponent` exists and should receive the callback. However, if any other listener modifies the state between these calls (e.g., PlaybackEngine rebuilding tracks), the indices could desync.

#### Cause 3: `groupTracks()` fires both `trackAdded` AND `tracksGrouped` 
In `groupTracks()`:
```cpp
listeners.call(&Listener::trackAdded, firstIndex, groupTrack);  // partial UI update
// ... set parentGroupIds ...
listeners.call(&Listener::tracksGrouped);  // nuclear rebuild
```
The `trackAdded` call inserts an extra lane at `firstIndex` and shifts all subsequent lane indices. Then `tracksGrouped` destroys everything and rebuilds. This double-firing is wasteful and could cause subtle issues with listener registration ordering.
